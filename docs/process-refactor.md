# Process Refactor: Task Recovery & Visibility

## Objective
Enhance the system's ability to cleanly recover from stuck, failed, or orphaned tasks (especially those with downstream dependencies), and surface these blocked states prominently in the Web UI.

## 1. Web UI Visibility & Alerts

### 1.1. Global Alert Bell Icon
*   **Trigger Conditions:** The backend must aggregate project statuses and flag any project containing `failed` tasks, open `disputes`, or tasks with `failure_count >= 3`.
*   **Dynamic Visibility:** The bell icon in the TopBar must be conditionally rendered. It remains completely hidden when all systems are healthy.
*   **Visual Styling:** When activated, the icon should display prominently (e.g., using a red color `text-red-500` or a red dot badge) to indicate attention is required.
*   **Interaction:** Tapping the bell icon must trigger a React Router navigation event directly to the Projects Dashboard (`/projects`), acting as a quick shortcut.

### 1.2. Global Daemon Status Indicator
*   **System Paused State:** If the user toggles "Stop Steroids" (setting `is_active = false`), the Web UI must prominently display a global "Daemon Paused" or "System Stopped" indicator in the TopBar or Sidebar.
*   **User Clarity:** This prevents user confusion where pending tasks exist but nothing is being processed due to the daemon's manual pause.

### 1.3. Project Dashboard Highlighting
*   **State Mapping:** The frontend project object must consume a boolean flag (e.g., `isBlocked` or `hasIssues`) provided by the backend's updated dynamic statistics endpoint.
*   **Conditional Formatting:** 
    *   Apply a red border (`border-red-500`) to the main `ProjectCard` container.
    *   Change the project title text color to red.
*   **Accessibility:** Ensure aria-labels or tooltips indicate *why* the project is highlighted (e.g., "Project contains failing tasks").

### 1.4. Project Card Task Counts (In Progress / Review)
*   **Component Consolidation:** Replace the individual "In Progress" and "In Review" statistics tiles on the project card with a single, compact UI element.
*   **Formatting Details:**
    *   Structure the text exactly as `[In Progress] / [In Review]` (e.g., `2/3`).
    *   Apply a green CSS class (e.g., `text-green-500`) to the "In Progress" integer.
    *   Apply an orange/amber CSS class (e.g., `text-orange-500`) to the "In Review" integer.

### Acceptance Criteria (Section 1)
- [x] The global alert bell icon is visible ONLY if at least one project has `failed` tasks, open `disputes`, or tasks with `failure_count >= 3`.
- [x] Clicking the global alert bell icon navigates the user to the `/projects` page.

## 2. CLI & Backend Recovery Mechanisms

### 2.1. Task State Synchronization (CLI Reset Command)
*   **Command Structure:** Add a new CLI subcommand `steroids tasks reset`. It must accept granular scoping flags: `--failed` (all failed), `--disputed` (all disputed), `--all` (both), or accept a specific `taskId`.
*   **Atomic Database Transaction:** The reset operation must be executed as a single SQLite transaction to prevent partial state corruption. The transaction must perform the following:
    1.  **Tasks Table:** Update the target tasks to `status = 'pending'`, `rejection_count = 0`, and `failure_count = 0`.
    2.  **Disputes Table:** Auto-resolve any `open` disputes tied to these tasks. Set `status = 'resolved'`, `resolution = 'custom'`, and `resolution_notes = 'Bulk reset via CLI'`.
    3.  **Locks Table:** Hard-delete any rows in `task_locks` that correspond to the reset tasks to ensure dead runners aren't holding the state hostage.
    4.  **Audit Table:** Insert a clear audit log entry specifying that a human-initiated bulk reset occurred.

### 2.2. Dependency Resolution verification
*   **Commit Validation:** During the reset or prior to a runner picking up the reset task, the system must verify the `reference_commit` of the task's upstream dependencies.
*   **Workspace Integrity:** Ensure that the base branch/commit assigned to the task actually contains the filesystem artifacts generated by its prerequisites. The orchestrator should perform a pre-flight check (e.g., via `git ls-tree <commit> <expected_path>`) to ensure the required foundation files exist before spawning a runner.
*   **Untracked Dependency Validation:** The pre-flight check must also account for untracked but necessary files left in the workspace from recent uncommitted upstream tasks, ensuring the runner has a functionally complete environment.

### Acceptance Criteria (Section 2)
- [x] A new CLI command `steroids tasks reset` is available and accepts `--failed`, `--disputed`, `--all`, or a positional `<taskId>` argument. It operates on the current working directory's project.
- [x] Running the reset command executes an atomic transaction that sets targeted task `status` to `pending` and resets both `rejection_count` and `failure_count` to 0.
- [x] The reset command automatically resolves any open disputes attached to the targeted tasks with the resolution text "Bulk reset via CLI".
- [x] The reset command clears any existing `task_locks` for the targeted tasks.
- [x] A detailed audit log entry is successfully recorded documenting the manual bulk reset.
- [x] The backend/orchestrator performs a pre-flight `git ls-tree` (or equivalent) check on the task's assigned `reference_commit` to verify the existence of upstream dependency files before a runner begins work.
- [x] The pre-flight check correctly accounts for and validates the presence of untracked files required by the task dependencies.
- [x] **Adversarial Review:** A Gemini adversarial review has been executed against the implementation commit(s) and returned a PASS contract.

## 3. Project Statistics Architecture Refactor

### 3.1. Dynamic Data Loading (On-Demand)
*   **Remove Global Caching:** Deprecate and remove the `pending_count`, `in_progress_count`, `review_count`, and `completed_count` columns from the `projects` table in the global home database (`~/.steroids/steroids.db`).
*   **Live Querying:** Refactor the `GET /api/projects` endpoint. For every registered project, the backend must dynamically open a read-only SQLite connection to the child project database (`.steroids/steroids.db`) and execute a `GROUP BY status` aggregate query. (Note: The API's `getProjectLiveData` already does some of this, but it needs to be made comprehensive).
*   **Daemon Performance Consideration:** Since the daemon relies on finding projects with pending tasks, removing global caching means the daemon (`src/runners/wakeup.ts`) must also query the child databases every minute. Given SQLite's speed (and WAL mode), this O(N) operation across registered projects is acceptable but must be implemented with efficient, short-lived read connections to prevent file-lock contention.
*   **No Repetitive Polling:** The Web UI must only fetch this data on explicit page load or user-initiated refreshes. Remove any frontend `setInterval` or polling mechanisms tied to global statistics.

### 3.2. Comprehensive Metrics
*   **Expanded Payload:** The dynamic query must map all internal SQLite statuses to the API response, explicitly adding counts for `failed` and `disputed` tasks.
*   **Blocked State Logic:** The backend should calculate an `isBlocked` boolean (true if `failed > 0` or `disputed > 0`) and pass it to the frontend to drive the Dashboard Highlighting logic (Section 1.2).

### Acceptance Criteria (Section 3)
- [x] The global home database (`~/.steroids/steroids.db`) no longer relies on duplicated `pending_count`, `in_progress_count`, `review_count`, or `completed_count` columns.
- [x] The `/api/projects` endpoint successfully fetches and aggregates task statuses dynamically from each individual project's database.
- [x] The daemon's 1-minute wakeup loop successfully queries child databases to discover pending work without causing file-lock contention.
- [x] The API response payload accurately includes counts for `failed` and `disputed` tasks.
- [x] The Web UI loads project statistics strictly on demand (page load/refresh) without relying on background interval polling (`setInterval`).
- [x] The backend API accurately returns an `isBlocked` boolean property (true when `failed` > 0 or `disputed` > 0) to drive the frontend UI highlighting.

## 4. Provider Error Resilience & Hibernation

### 4.1. Progressive Hibernation & Lightweight Pinging (Token/Capacity Limits)
*   **Unified Limit Handling:** Whether the error is a hard limit (e.g., "Out of Credits", "Token Exhaustion") or a soft limit (e.g., "High Demand", 503s), the system will treat it as a capacity issue that might resolve over time (e.g., credits topped up, demand drops).
*   **Progressive Backoff Schedule:** When an invocation fails due to capacity/tokens, the project enters a `hibernating` state.
    *   **Attempt 1:** Wait 5 to 10 minutes.
    *   **Attempt 2+:** Poll every 30 minutes indefinitely (or until manual intervention).
*   **Deterministic Lightweight Pinging:** Crucially, the retry mechanism must **not** attempt to spawn the full, heavy AI orchestrator or resend the massive prompt context. This wastes tokens and risks compounding failures.
    *   Instead, the daemon's regular event loop will execute a deterministic, lightweight "ping" to the API provider.
    *   This ping should be a minimal, stateless request (e.g., sending the prompt "hello world!" using the configured provider).
*   **Resume on Ping Success:** If the lightweight ping returns a successful `200 OK` response, the system definitively knows the API capacity has been restored. Only at this moment does the daemon clear the hibernation state and resume normal execution (spawning the full agent where it left off).

### 4.2. Hard Failures (Safety & Policy Violations)
*   **Orchestrator Intervention:** Errors explicitly identified as "Safety Violations", "Policy Violations", or "Invalid Prompts" by the provider (e.g., OpenAI/Codex flagging the prompt) must **not** enter the hibernation loop. Instead, the system must first pass the error and context back to the orchestrator.
*   **Attempt Resolution (Bounded):** The orchestrator will attempt to analyze the violation and suggest a resolution (e.g., automatically rephrasing the prompt or adjusting the specification to comply with provider policies). This loop must be strictly bounded to a maximum of 2 rewrite attempts to prevent an infinite cycle of violating prompts.
*   **Full Stop & User Notification:** If the orchestrator cannot resolve the violation after the maximum attempts, or determines that the task fundamentally conflicts with impassable provider policies, the system must put a "full stop" to the task. It will immediately fail the task (or open a `steroids dispute`) and explicitly inform the user of the hard block.

### Acceptance Criteria (Section 4)
- [ ] A project-level hibernation state is established to track the backoff schedule.
- [ ] Capacity and token exhaustion errors trigger the progressive backoff schedule (5-10m -> 30m continuous).
- [ ] During the backoff schedule, the daemon does **not** spawn the AI orchestrator or send full project prompts.
- [ ] The daemon executes a lightweight, deterministic "ping" (e.g., sending "hello world!") to the provider API when a backoff timer expires.
- [ ] The full orchestrator is only re-engaged after the lightweight ping returns a successful response.
- [ ] Explicit Safety/Policy violation errors bypass the hibernation loop.
- [ ] Upon a Safety/Policy violation, the orchestrator is invoked to attempt an automatic resolution of the prompt/specification.
- [ ] If the orchestrator cannot resolve the violation, the task immediately enters a failed/disputed state and surfaces a clear notification to the user.
- [ ] **Adversarial Review:** A Gemini adversarial review has been executed against the implementation commit(s) and returned a PASS contract.

## 5. Lost Session Resilience (Fallback to Fresh)

### 5.1. Auto-Fallback on Resume Failure
*   **Error Catching:** Currently, tasks fail immediately if a provider CLI returns a "session not found" error (e.g., `gemini --resume <session_id>` fails when local `.gemini/state.json` is missing or runner is on a different machine). 
*   **Orchestrator Responsibility:** The orchestrating runner classes (`src/runners/coder.ts`, `src/runners/reviewer.ts`) are solely responsible for catching these explicit "resume failed" or "session not found" errors returned by the lower-level API providers.
*   **Fresh Invocation Generation:** Instead of failing the task, the orchestrator should catch the error, discard the `resumed_from_session_id`, and immediately invoke the provider again with `invocation_mode = 'fresh'`. 
*   **Context Injection:** The fresh invocation will naturally rebuild the full conversation history from the database's `task_invocations` table and inject it as a single monolithic prompt, effectively rebuilding the lost session context.

### Acceptance Criteria (Section 5)
- [x] Provider wrappers explicitly throw a classified `SessionNotFoundError` when a provider CLI refuses to resume a session.
- [x] The orchestrating runners (`coder.ts`, `reviewer.ts`) catch `SessionNotFoundError`.
- [x] Upon catching the error, the orchestrating runner automatically reconstructs the full prompt history from the database and retries the API call using `invocation_mode: 'fresh'` instead of failing the task.

## 7. Daemon Control & Wakeup Logic

### 7.1. Refactored "Stop/Start Steroids" Toggle
*   **Stateful Polling:** Re-architect the `/api/runners/cron/stop` and `start` logic. Instead of aggressively uninstalling the cron job or killing the node process, the toggle should simply flip an `is_active` boolean state flag.
*   **Global State Storage:** To persist across daemon restarts, this `is_active` flag must be stored in the `_global_schema` table of the global database (`~/.steroids/steroids.db`).
*   **Idle Loop:** The daemon process must remain alive, ticking every minute. On each tick, it checks the `is_active` flag. 
    *   If `true`, it proceeds to query projects and wake up runners. 
    *   If `false`, it logs an idle tick and sleeps for another minute, doing nothing else.

### 7.2. New "Wake Up Runners" Button
*   **UI Placement:** Add a prominent "Wake Up Runners" button in the `Sidebar.tsx`, positioned immediately above the Stop/Start toggle.
*   **Manual Trigger:** Create a new API endpoint (e.g., `POST /api/runners/wakeup/now`). When clicked, this endpoint signals the running daemon to bypass its 1-minute sleep interval and execute an immediate wakeup cycle, providing instant feedback to the user.
*   **Single-Flight Mutex:** To prevent race conditions, the daemon must implement an in-memory lock or mutex around the wakeup cycle. If the user clicks the manual wakeup button while the background 1-minute tick is already executing a wakeup, the manual trigger must immediately return a `429 Too Many Requests` or gracefully wait without spawning duplicate runner processes.

### Acceptance Criteria (Section 7)
- [ ] Toggling "Stop Steroids" sets an internal `is_active = false` flag in the global database's `_global_schema` table rather than uninstalling the cron job or killing the process outright.
- [ ] When `is_active` is false, the daemon continues ticking every minute but completely skips triggering any runner wakeup logic.

## 8. Architectural Safeguards & Cleanups

### 8.1. Race Condition Prevention (Task Reset)
*   **Active Runner Termination:** When executing the `steroids tasks reset` command, simply deleting database locks is insufficient and introduces race conditions.
*   **Process Management:** Before clearing locks and resetting status, the command must explicitly look up the task's active runner PID in the global registry and forcefully kill the process to ensure it cannot overwrite the reset state.
*   **PID Reuse Sanity Check:** Ensure the system validates that the PID still belongs to a Steroids runner process (e.g., checking process name or start time) to avoid inadvertently killing an unrelated OS process if the PID was reused.

### 8.2. Token Guard (Lost Session Fallback)
*   **Monolithic Prompt Validation:** When rebuilding a lost session as a "fresh" monolithic prompt, the accumulated context (especially after many tool calls/rejections) poses a high risk of exceeding provider token limits (e.g., HTTP 413 Payload Too Large).
*   **Token Calculation & Model Lookup:** The orchestrator must accurately figure out the actual payload size. This involves taking the selected model identifier and dynamically looking up its maximum context window limits.
*   **Reserved Output Headroom:** The calculation must explicitly reserve a mandatory token budget (e.g., 4000-8000 tokens depending on the model) for the provider's *response* and tool executions. The input prompt cannot consume 100% of the context window.
*   **Guaranteed Preservation:** Truncation logic must **prioritize preserving the System Prompt and Task Specification** above all else to ensure AI performance is not degraded (a "bad fallback").
*   **Truncation/Guard Logic:** The orchestrator must implement a "Token Guard" prior to fallback. If the reconstructed history plus the required response headroom exceeds the safe context limits for that specific model, it must selectively prune older tool execution outputs or intermediate thought blocks. If the required context (System Prompt + Task Spec) *alone* exceeds the limit, it must deterministically hard-fail with a "Context Too Large" error.

### 8.3. O(N) Querying Timeouts & UI Disabling
*   **Connection Resilience:** The shift to dynamic, real-time querying of child databases for statistics/wakeup checks (Section 3.1) introduces risk if a single project's drive is slow or disconnected.
*   **Strict Timeouts:** Implement a strict connection timeout (e.g., 500ms) for each database query inside the aggregation loop. Unresponsive projects must be gracefully skipped rather than allowed to hang the entire API request or daemon wakeup cycle.
*   **File Descriptor Exhaustion:** To support long-term sustainability (weeks of uptime), the daemon must immediately close the SQLite connection to the child database the moment the aggregation query completes, ensuring file limits are never reached.
*   **UI Feedback (Greyed Out):** If the daemon or API cannot reach a project's database due to a timeout, that project should be visually greyed out on the Projects Dashboard. Its project card should be disabled and unclickable to explicitly indicate to the user that the project is currently unreachable.

### 8.4. UI Redundancy Cleanup
*   **Remove "Tasks" UI Page:** The dedicated "Running Tasks" page and its accompanying `/tasks` route are redundant.
*   **Sidebar Removal:** The "Tasks" navigation link must be entirely removed from the left-hand menu (`Sidebar.tsx`).

### 8.5. Worktree / Clone Reuse Guarantee (Parallel Mode)
*   **Architectural Clarification (Clones vs. Worktrees):** While Steroids documentation refers to parallel execution environments as "worktrees", the system does **not** use Git's native `git worktree add` feature. Instead, Steroids implements isolated workspaces using **`git clone --local`** (creating full, separate `.git` directories at `~/.steroids/workspaces/`). This custom approach ensures absolute dependency isolation (e.g., stopping Node.js or Python from traversing up into a shared `node_modules` or `venv` folder).
*   **The Goal:** If a task in a parallel execution mode fails and is restarted via the `reset` command, the system MUST return to the exact same isolated workspace (clone) where the failure occurred. Uncommitted files or partial coder turns must not be lost by blindly spawning a new session.
*   **Global DB Reconciliation:** To guarantee this, the `steroids tasks reset` command must perform the following actions in the global database (`~/.steroids/steroids.db`):
    1.  **Locate Workstream (Deterministic Lookup):** The system must look up the exact workstream by querying the task's last known `runner_id` from the `task_invocations` table, and joining that against the global `workstreams` table. Relying solely on `section_id` is ambiguous because a section might be split across multiple workstreams.
    2.  **Revoke Lease:** Clear the `runner_id` and `lease_expires_at` on that specific workstream to mark it as available for a new runner.
    3.  **Unblock Session:** If the parent `parallel_session` is in a blocked or failed state (e.g., `blocked_recovery`, `failed`), update its status back to `running`.
*   **Daemon Resurrection:** Because the session is now `running` and the workstream has no active lease, the background daemon's normal `wakeup` loop will automatically spawn a new runner with `--project <existing_clone_path>`. The new runner will boot up, find the `pending` task, and immediately resume work in the exact same directory, preserving all uncommitted changes.
